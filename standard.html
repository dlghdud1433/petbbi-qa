🎯 문제 정의

Petbbi는 2주 스프린트로 운영되며 별도 QA팀이 없어 PM이 테스트를 직접 설계한다. 기획서는 "정상 동작"만 기술하는 구조라 매 스프린트마다 아래 3가지 문제가 반복된다.

#문제비즈니스 영향
1 누락 — 권한 거부·네트워크 끊김·빈 상태 같은 엣지케이스가 매번 빠짐 프로덕션 크래시 → 앱스토어 평점 하락
2 기준 부재 — P0/P1/P2 판단 기준이 문서화되지 않아 출시 직전 "무엇부터 막을지" 혼선 릴리즈 지연 · 출시 후 핫픽스
3 반복 공수 — 새 기획서가 나올 때마다 처음부터 손으로 케이스를 다시 작성 QA 시간 부족 → 커버리지 저하
해결 목표
기획서(.md)를 입력하면 Happy/Unhappy/Edge 케이스 자동 생성 + P0/P1/P2 점수화 기준 적용 + 섹션별 커버리지 자동 검증까지 한 번의 파이프라인으로 완결되는 재현 가능한 도구 구축

# 새 기획서가 들어올 때마다 (Claude Code 터미널)
# 1. 새 기획서 .md 를 data/ 폴더에 교체
/analyze # 기능 섹션·세부항목 추출, 빠진 엣지 후보 진단
/insight # Happy/Unhappy/Edge + P0/P1/P2 기준 적용
/generate # qa-checklist.csv 자동 생성 (9컬럼 고정)
/review # 섹션 커버리지·P0 포함·엣지 유형 점검
node scripts/coverage_report.js # 섹션별 커버리지 리포트 자동 출력
체크리스트 양식 설계 근거 (본인 설계) QA 담당자는 "어떤 케이스를 먼저 실행할지"를, PM은 "출시를 막을 P0가 무엇인지"를 본다. 컬럼을 테스트유형·우선순위·출처 3축으로 고정해 두 관점을 한 표에서 필터링할 수 있게 했다. 특히 출처(기획서기반/AI추가) 컬럼은 "기획서 자체의 빈틈"을 가시화해 다음 스프린트 기획서 작성 품질로 피드백된다. coverage_report.js는 2주마다 갱신되는 기획서 환경에서 수동 집계를 없애기 위해 추가했다.

🔧 트러블슈팅 기록
① CSV 인코딩 문제
현상
Windows 환경에서 생성된 CSV에 UTF-8 BOM이 포함되어 파싱 시 첫 컬럼명(기능섹션)이 깨지는 문제 발생
해결
charCodeAt(0) === 0xFEFF 로 BOM 자동 감지·제거 + 콘솔 경고로 원본 파일 수정 유도
② 쉼표 포함 필드 파싱 오류
현상
기대결과 컬럼에 쉼표가 포함된 셀이 분리되어 컬럼 수 불일치 발생
해결
단순 split(',') 대신 RFC4180 방식의 큰따옴표 인식 완전 파서 구현 ("" 이스케이프, 필드 내 줄바꿈까지 처리)
③ 파이프라인 경고 미감지 문제
현상
/review 실행 시 경고가 있어도 종료 코드 0으로 나와 자동화 환경에서 경고를 놓칠 수 있음
해결
경고 발생 시 process.exitCode = 1 로 설정하여 CI/CD 파이프라인에서 차단 조건으로 활용 가능하도록 처리
